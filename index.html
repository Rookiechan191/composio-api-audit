How the research got done

Search and generation were split on purpose — a bundled search+generation call blew through free-tier token budgets in one shot.

01

Search

Tavily API pulls the top results for each app's docs, auth, and developer access pages. Free tier, no card.

Composio SDK

Cross-checking against Composio's own toolkit registry

Pass 1 used generic search + extraction (Tavily + Groq) rather than Composio's SDK, to demonstrate a framework-agnostic agentic loop. But there's a more direct use of Composio's own tooling: their toolkit registry is the actual ground truth for "does an agent-callable integration already exist" — more authoritative than any web search, since Composio is the source of it.

composio_toolkit_check.py — live results

What it does Calls composio.toolkits.list() to fetch Composio's full toolkit registry, fuzzy-matches each of the 100 app names against it, and compares the result against what Pass 1's agent guessed for existing_mcp — flagging any disagreement.
Status run live Actually run against the live Composio API — 1,249 toolkits pulled from their real registry, cross-checked against all 98 valid Pass 1 records. Result: 31 disagreements (32%). Splitting those apart: 27 were Pass 1 being too conservative — Composio already has a toolkit for the app (e.g. Pipedrive, Discord, Jira, Asana, QuickBooks) but the agent's web search couldn't confirm it and marked it "unknown" or false. Only 4 were the agent overclaiming — saying an MCP existed when Composio's registry shows it doesn't (Ecwid, Vercel, NotebookLM, Otter AI).
Headline finding 27 of 98 apps already have a Composio toolkit that Pass 1's web search missed entirely — zero extra research needed to build agents for those, just connect and go. The research agent under-reports MCP coverage rather than fabricating it, which is the safer failure mode, but it means existing_mcp in the main table below should be read as a floor, not a ceiling — Composio's own registry is the better source of truth for that specific field.
Pattern 02

Verification pass — what a second look caught

Three separate loops: an automated consistency pass (106 flags across 100 records, mostly mechanical), an independent hand fact-check against real docs on a 10-app sample, and a Composio SDK cross-check against their own toolkit registry (see below). The hand check found real errors the automated pass missed entirely.

    Live fact-check: 6 / 10 correct on hand sample

    Method 10 records (10% of the dataset) independently cross-checked against real developer docs by hand: Ramp, Attio, Devin, GitHub, WhatsApp Business, Stripe, iPayX, Mermaid CLI, Otter AI, Ahrefs.
    Ahrefs wrong Record marked almost every field "unknown," including an invalid verdict value. Ahrefs actually has a full documented API v3, a free no-subscription "Public" endpoint tier, and an official hosted MCP server. The original search step simply failed to surface Ahrefs's own developer docs.
    Otter AI wrong Record claimed self-serve, free public API. Otter's own help center says the public API is Enterprise-only, gated behind contacting an account manager.
    Mermaid CLI wrong Record described a REST API with OAuth2 and admin approval. That's actually the API for an unrelated coral-reef data nonprofit that happens to share the name "MERMAID." The real app — the diagramming CLI tool — has no REST API at all; it's a local command-line renderer.
    iPayX wrong, excluded from stats No company matching this name turned up in an independent web check. Its evidence URL was identical to the unrelated record above it — a strong sign of a recycled or fabricated result.
    Ramp, Attio, Devin, GitHub, WhatsApp Business, Stripe correct All six matched real docs on access type and buildability. Devin's plan-name detail was slightly off (says "Enterprise only"; the $500/mo Team plan already includes API access) but the gated_paid verdict itself held up.
